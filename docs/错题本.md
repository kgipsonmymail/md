## Bug #3: 环境变量配置暴露

### 问题描述

敏感配置信息（如 API Key）直接暴露在前端代码中，存在安全风险。

### 复现步骤

1. 查看前端源代码
2. 发现 API Key 等敏感信息

### 根本原因

- 配置信息硬编码在前端
- 没有使用环境变量
- 缺少安全意识

### 解决方案

1. **使用环境变量**

   - 将敏感配置存放在 .env 文件
   - 使用 Vite 的环境变量机制
   - .env 文件不提交到 Git

2. **后端代理**

   - 将敏感操作放在后端
   - 前端通过 API 调用
   - 避免暴露敏感信息

3. **隐藏配置界面**
   - 移除"显示配置"按钮
   - 通过环境变量控制
   - 不暴露给用户

### 实施步骤

1. 创建 .env 文件
2. 配置环境变量
3. 修改代码读取环境变量
4. 移除前端配置界面
5. 更新 .gitignore

## Bug #4: 渲染结果与公众号不一致

### 问题描述

在编辑器中预览的效果与复制到公众号后的效果不一致。

### 复现步骤

1. 在编辑器中排版文章
2. 预览渲染效果
3. 复制到公众号编辑器
4. 效果不一致

### 根本原因

- 公众号编辑器对某些 CSS 样式有限制
- 部分扩展语法不被支持
- 图片链接格式不兼容

### 解决方案

1. **使用内联 CSS**

   - 使用 juice 库将 CSS 内联
   - 确保样式在公众号中生效
   - 测试常用样式

2. **限制扩展语法**

   - 只使用公众号支持的语法
   - 提供兼容性说明
   - 警告不支持的语法

3. **图片链接格式**
   - 使用 CDN 加速链接
   - 确保链接可访问
   - 测试图片显示

### 测试方法

1. 在公众号编辑器中测试
2. 使用手机预览
3. 对比不同平台效果

## Bug #5: AI API 调用失败

### 问题描述

调用 AI API 时偶尔失败，返回错误信息。

### 复现步骤

1. 输入文章内容
2. 调用 AI 排版
3. 偶尔失败

### 错误信息

```
Error: Request failed with status code 429
```

### 根本原因

- API 调用频率限制
- 网络不稳定
- API Key 过期

### 解决方案

1. **添加重试机制**

   - 失败后自动重试
   - 指数退避策略
   - 最多重试 3 次

2. **错误处理**

   - 捕获所有错误
   - 显示友好的错误提示
   - 提供解决方案

3. **监控和日志**
   - 记录 API 调用日志
   - 监控成功率
   - 及时发现问题

### 代码示例

```javascript
async function callAIWithRetry(prompt, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      const result = await callAI(prompt)
      return result
    }
    catch (error) {
      if (i === maxRetries - 1)
        throw error
      await sleep(2 ** i * 1000) // 指数退避
    }
  }
}
```

2. **错误处理**

   - 捕获上传错误
   - 显示友好的错误提示
   - 提供解决方案

3. **分片上传**
   - 大文件分片上传
   - 支持断点续传
   - 提高上传成功率

### 代码示例

```javascript
async function compressImage(file, maxSizeMB = 5) {
  const options = {
    maxSizeMB,
    maxWidthOrHeight: 1920,
    useWebWorker: true
  }
  try {
    const compressedFile = await imageCompression(file, options)
    return compressedFile
  }
  catch (error) {
    console.error('压缩失败:', error)
    return file
  }
}
```

## Bug #7: 编辑器内容丢失

### 问题描述

用户在编辑器中输入的内容意外丢失。

### 复现步骤

1. 在编辑器中输入内容
2. 刷新页面或关闭标签
3. 内容丢失

### 根本原因

- 没有自动保存
- 浏览器缓存未使用
- 用户误操作

### 解决方案

1. **自动保存**

   - 定期自动保存到 localStorage
   - 显示保存状态
   - 支持恢复

2. **离开确认**

   - 页面关闭前确认
   - 提示用户保存
   - 防止误操作

3. **版本历史**
   - 保存多个版本
   - 支持回滚
   - 查看历史记录

### 代码示例

```javascript
// 自动保存
setInterval(() => {
  const content = editor.getValue()
  localStorage.setItem('editor-content', content)
  showSaveStatus('已自动保存')
}, 30000) // 每30秒保存一次

// 离开确认
window.addEventListener('beforeunload', (e) => {
  if (hasUnsavedChanges) {
    e.preventDefault()
    e.returnValue = '您有未保存的内容，确定要离开吗？'
  }
})
```

## Bug #8: HTML 颜色解析失效

### 问题描述

在转换某些微信公众号文章时，特定的文字颜色（如橙色 `#DD7801`）无法被正确解析，转换后的 Markdown 中呈现为灰色 `#555555`。

### 复现步骤

1. 导入包含复杂样式的微信公众号文章 HTML。
2. 观察转换结果中的颜色标签。
3. 发现颜色不是预期的橙色。

### 根本原因

- 微信 HTML 中存在重复的 `color` 属性（例如 `style="color: #555555;color: rgb(222,120,2)!important;"`）。
- 原解析器使用简单的正则表达式 `style.match(/color:\s*([^;]+)/i)`，仅能匹配到第一个出现的颜色。
- 未处理 CSS 优先级规则（如 `!important`）及属性覆盖逻辑。

### 解决方案

1. **引入通用样式提取逻辑**：
   在 `HtmlParser` 中增加 `getStyleValue` 方法。
2. **支持优先级与去重**：
   - 使用正则表达式匹配所有的属性声明。
   - 记录最新定义的属性值。
   - 优先识别并保留带有 `!important` 标记的属性。
   - 自动清理 `!important` 后缀以保证 Markdown 渲染器的兼容性。
3. **全面应用**：
   将此逻辑应用到 `color`、`font-size`、`background` 及 `text-align` 的解析中。

### 预防措施

- 对于 CSS 内联样式的解析，应避免使用简单的正则全局匹配，而应遵循 CSS 级联规则。
- 增加对复杂 HTML 样式的测试用例。
